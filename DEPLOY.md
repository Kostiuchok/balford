# Автосинхронізація теми з сервером

Зараз деплой теми — вручну через FTP, файл за файлом (див. `README.md` →
"Локальна розробка"). Це нормально для разових правок, але при ітеративних
CSS/JS-правках (кілька раундів "поправив → залив → перевірив") набридає.

Нижче — два варіанти автозаливання файлів на стейджинг одразу при
збереженні, без ручного FTP щоразу. Обидва працюють лише з вашого боку:
креденшели лишаються у вашому клієнті, нікуди більше не передаються.

**Дані підключення** (з `README.md` → "Хостинг і доступи"):
- Хост: `wvh1.mirohost.net`
- Логін: `balford`
- Пароль: у вас (менеджер паролів / приватний канал)
- Протокол: **звичайний FTP, без TLS** (сервер не підтримує AUTH SSL/TLS)
- Шлях на сервері: `balford.com.ua/wp-new/wp-content/themes/balford/`
  (FTP root уже chroot-нутий у `/`, тобто після підключення ви одразу в
  корені акаунта — не `/var/www/...`)

---

## Варіант 1: WinSCP — "Keep remote directory up to date"

1. Відкрити WinSCP → New Session.
2. File protocol: **FTP**, Encryption: **No encryption** (сервер не
   підтримує TLS).
3. Host name: `wvh1.mirohost.net`, Port: `21`.
4. User name: `balford`, Password: ввести свій (не зберігати в plain-text
   файлах репо).
5. Login → підключення відкриється в корені акаунта (`/`). Перейти в
   `balford.com.ua/wp-new/wp-content/themes/balford/`.
6. У локальній панелі перейти в
   `C:\PROJECTS\balford\balford.com.ua\wp-content\themes\balford\`.
7. Меню **Commands → Keep Remote Directory Up To Date** (або
   `Ctrl+Alt+U`).
8. У вікні, що відкриється:
   - переконайтесь, що локальна й віддалена директорії — саме ті, що
     вище (тема, не корінь репо — інакше почне заливати `legacy-static/`
     і `.git/`, які на сервері не потрібні);
   - увімкнути **"Synchronize automatically"** (моніторить файлову
     систему й заливає зміни одразу при збереженні файлу);
   - опційно: **"Show synchronization dialog before uploading"** —
     якщо хочете підтверджувати кожен раз, а не заливати наосліп.
9. Натиснути **Start**. Лишити WinSCP відкритим у фоні — поки сесія
   активна, кожне збереження файлу в темі автоматично летить на
   `wp-new/`.

Зберегти сесію (Save) з цими налаштуваннями, щоб наступного разу не
налаштовувати заново — WinSCP пропонує зберегти пароль у власному
зашифрованому сховищі сесій (не в текстовому файлі проєкту).

## Варіант 2: VS Code — розширення SFTP (liximomo)

1. Встановити розширення **"SFTP" (liximomo)** з маркетплейсу VS Code.
2. У корені проєкту: `Ctrl+Shift+P` → **"SFTP: Config"** — створить
   `.vscode/sftp.json`. Цей файл вже виключений з git (`.vscode/` є в
   `.gitignore`), тож пароль там зберігати безпечно для цього репо.
3. Заповнити `.vscode/sftp.json`:

   ```json
   {
     "name": "balford-staging",
     "host": "wvh1.mirohost.net",
     "protocol": "ftp",
     "port": 21,
     "username": "balford",
     "password": "ВАШ_ПАРОЛЬ",
     "remotePath": "/balford.com.ua/wp-new/wp-content/themes/balford",
     "uploadOnSave": true,
     "watcher": {
       "files": "wp-content/themes/balford/**/*",
       "autoUpload": true,
       "autoDelete": false
     }
   }
   ```

   Розширення саме прив'яже `remotePath` лише до піддиректорії теми —
   редагувати варто тільки файли всередині
   `wp-content/themes/balford/`, інше на сервер не заливається.
4. Зберегти файл. При наступному збереженні будь-якого файлу теми в VS
   Code розширення само підключиться й закине його на сервер
   (`uploadOnSave: true`).
5. Статус заливання видно в нижній панелі VS Code (SFTP output) —
   корисно, якщо потрібно переконатись, що файл дійсно долетів, перед
   тим як перевіряти на стейджингу.

## Застереження

- Обидва варіанти заливають **тільки** `wp-content/themes/balford/`.
  Ядро WP, плагіни, `wp-config.php` — поза скоупом і мають лишатись
  поза автосинком (вони й поза git з тих самих причин, див.
  `README.md`).
- FTP тут без TLS — пароль передається у відкритому вигляді. Обидва
  клієнти варто запускати лише в довіреній мережі (не з публічного
  Wi-Fi), як і зазначено в `README.md`.
- Не комітьте `.vscode/sftp.json`, якщо коли-небудь приберете
  `.vscode/` з `.gitignore` для інших цілей — перевірте, що виключення
  лишилось.
