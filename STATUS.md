# STATUS — міграція balford.com.ua на WordPress

## Контекст проєкту
Мігруємо зі статичного сайту (`legacy-static/`) на WordPress — заради зручності
нетехнічного редактора контенту. Стейджинг: **https://balford.com.ua/wp-new/**
Мультимовність UA/EN — через **Polylang**, SEO — через **Yoast SEO**.
Планована версія PHP — **8.5**.

Хостинг: Mirohost, FTP-хост `wvh1.mirohost.net` (раніше фігурував як
`fvh16.mirohost.net` — актуальний саме `wvh1`), FTP root вже chroot-нутий
у домашню директорію (`/`, а не `/var/www/balford`), усередині — папка
`balford.com.ua/`, у ній `wp-new/`. Логін `balford`. FTP без TLS (сервер не
підтримує AUTH SSL/TLS — пароль передається у відкритому вигляді). Пароль
користувач тримає в себе, в цьому файлі не зберігається.

## ✅ Вирішено: версія PHP
PHP на сервері оновлено до **8.5.8** (підтверджено діагностикою 22.07.2026) —
ядро WordPress тепер вантажиться без fatal-помилок несумісності.

## Деталі перенесення контенту (22.07.2026)

**Сторінки** (slug у UA лишився як в оригіналі, EN — довелось узяти окремі
слаги, бо Polylang НЕ дозволяє однаковий slug для двох мов "з коробки" —
`wp_unique_post_slug` не має спеціального винятку для мовних пар у цій
версії Polylang, тому однакові slug призводили до автоматичного `-2`):
- `about` (uk) / `about-company` (en) — контент з блоку "Про компанію"/
  "About the company" на старій головній
- `ecology` (uk) / `ecology-impact` (en) — заголовок сторінки = сам текст
  висновку ОВД (щоб коректно підтягувався в тизер на головній через
  `get_the_title()`), контент — параграф + посилання на PDF-висновок
  (зовнішній, на eia.menr.gov.ua, як в оригіналі)
- `contacts` (uk) / `contact-us` (en) — телефон/адреса/email з футера
  легасі-сайту

Оскільки шаблон сторінки (`page-about.php` тощо) в темі вибирався
**виключно по slug**, а EN-сторінки мають інші slug — явно прописано
`_wp_page_template` postmeta для всіх 6 сторінок, щоб потрібний шаблон
підвантажувався незалежно від slug.

**Новини** (CPT `news`, slug новин отримав суфікс `-en` для англійської
версії, наприклад `vidkryly-buvet` / `vidkryly-buvet-en`): усі 8 новин
з легасі (vidkryly-buvet, remont-krovli, blagoustrii-dorogy,
vidnovlennya-mostu, ges-novogo-typu, nimeckyi-dosvid, norway-embassy,
povidomlennya-pro-pidpysannya-zvitiv) перенесені з оригінальними датами
публікації, текстом UA+EN і зображеннями (скопійовані сервером напряму з
`wp-content/themes/balford/assets/img/news/` у Media Library, без
повторного завантаження з локальної машини). Для новини
"povidomlennya-pro-pidpysannya-zvitiv" довантажено 2 PDF-додатки
(`legacy-static/news/12_05_2023/*.pdf`) в Media Library, посилання на них
вставлені в текст.

**Важливий технічний нюанс, який довелось виправляти:** кастомний тип
постів `news` за замовчуванням НЕ входив у список типів, які Polylang
перекладає (`$options['post_types']`, порожній масив, поки не увімкнено
явно) — через це архів `/news/` показував впереміш і UA, і EN новини,
а `/en/news/` віддавав 404. Виправлено додаванням `'news'` в
`PLL()->model->options['post_types']` + `flush_rewrite_rules(true)`.
Після цього `/news/` фільтрує лише українські, `/en/news/` — лише
англійські новини.

**Правки в темі (закомічені в репо):** `page-about.php`, `page-ecology.php`,
`page-contacts.php` — банер-заголовок сторінки був захардкоджений
українською (`<h4>Про компанію</h4>` тощо) незалежно від мови сторінки;
замінено на `<?php the_title(); ?>`, щоб банер показував título
відповідною мовою. `single-news.php` вже було коректно (використовував
`the_title()`).

## 🟡 Наступна сесія: перенесення на продакшн (на основний домен)

Власник плаує перенести сайт з `/wp-new/` на корінь `balford.com.ua/`
(замінивши поточний legacy-static). Чекліст для цього переносу:

1. **Бекап перед стартом:** зберегти поточні файли legacy-static з
   продакшн-кореня (вони й так є в цьому репо в `legacy-static/`, але
   варто перевірити, чи на сервері немає розбіжностей/новішого контенту
   перед видаленням).
2. **⚠️ Заший абсолютні URL `/wp-new/` у контенті новин.** Перевірено
   22.07.2026: **14 постів новин** (усі, окрім Dovge/Norway Embassy —
   у неї не було галереї) містять literal `https://balford.com.ua/wp-new/...`
   прямо в `post_content` — це URL зображень галереї (вставлені як
   `<img src="...">` при імпорті контенту) і 2 посилання на PDF-звіти в
   новині "ТОВ «Балфорд Україна» повідомляє" (uk+en). Це **не**
   динамічні виклики типу `get_template_directory_uri()`, а буквально
   застряглий текст. **Обов'язково зробити search-replace
   `/wp-new/` → `` (порожній рядок) по `wp_posts.post_content`** одразу
   після переносу — інакше ці зображення й посилання на PDF зламаються
   на проді. Можна через WP-CLI `search-replace`, плагін Better Search
   Replace, або прямий SQL `UPDATE wp_posts SET post_content =
   REPLACE(post_content, 'https://balford.com.ua/wp-new/', 'https://balford.com.ua/')`.
3. **Оновити `siteurl`/`home` в wp_options** з `.../wp-new` на
   `https://balford.com.ua` (Налаштування → Загальні в wp-admin, або
   прямо в БД) — критичний крок, впливає на всі динамічні посилання
   сайту (меню, permalink, зображення через `wp_upload_dir()` тощо,
   на відміну від пункту 2 вище, який саме про НЕ-динамічний застряглий
   текст).
4. **Перенести файли:** або перемістити вміст `wp-new/` в корінь
   `balford.com.ua/` (FTP rename/move), або новий чистий деплой у
   корінь + повторний імпорт контенту з нуля (перший варіант простіший
   і зберігає вже зроблену роботу).
5. **SEO-редиректи зі старих `.html`-адрес.** Старий сайт мав URL типу
   `/news-vidkryly-buvet.html`, новий WP — `/news/vidkryly-buvet/`.
   Якщо старі сторінки проіндексовані Google / мають зовнішні
   посилання — варто додати 301-редиректи зі старих шляхів на нові
   (через Yoast SEO Premium redirect manager, чи `.htaccess`/nginx
   правила, залежно від можливостей хостингу), інакше втратиться
   SEO-вага і з'являться 404 для старих посилань.
6. **Перевірити permalink-структуру після переносу** — зайти в
   Налаштування → Постійні посилання і натиснути "Зберегти" (форсує
   `flush_rewrite_rules()`), щоб мовні URL для CPT `news` (`/en/news/`)
   не почали 404-тити знову на новому шляху (це вже траплялось раз на
   стейджингу, див. вище).
7. **Повторити технічну перевірку** (усі сторінки — HTTP 200, обидві
   мови, зображення, хлібні крихти, перемикач мов) на продакшн-домені
   так само, як робилось на `/wp-new/`.
8. **SSL / DNS:** ймовірно, змін не потрібно (той самий домен, той
   самий хостинг-акаунт), але варто підтвердити, що HTTPS і сертифікат
   покривають кореневий шлях так само, як покривали `/wp-new/`.
